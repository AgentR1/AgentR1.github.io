P0 · 首次使用

快速开始

先确保依赖安装和 LLM 配置正确，再用示例数据完整走一遍预测任务。这个阶段的目标不是调到最优，而是理解系统如何运转。

建议阅读顺序：安装 → 第一个任务 → TUI 操作

安装与环境配置

CastClaw 的本体通过 npm 安装，但运行阶段通常会同时依赖 Bun、Python 与 uv。建议先把运行时准备齐，再启动 TUI 做环境验证。

Prerequisites

前置依赖

Bun ≥ 1.3.11 用于前端与脚本执行，Python ≥ 3.10 + uv 用于预测后端与依赖管理，外加至少一个可用的 LLM API Key。

Bun ≥ 1.3.11 Python ≥ 3.10 uv LLM API Key
Validation

建议验证项

先确认 CLI、Python 和 uv 均可被当前 shell 识别，再启动一次 CastClaw 检查是否能顺利进入 TUI，并在日志中看到模型配置被正常加载。

# 安装 CastClaw CLI
npm install -g castclaw

# 验证基础运行时
castclaw --version
python --version
uv --version

# 如果 Python 后端依赖尚未同步
cd python
uv sync

LLM 可以通过环境变量或 castclaw.json 提供。环境变量适合本机快速切换，配置文件适合团队共享一套默认策略。

# 任选其一或按需配置多个提供商
export ANTHROPIC_API_KEY=your_key_here
export OPENAI_API_KEY=your_key_here
export GOOGLE_GENERATIVE_AI_API_KEY=your_key_here
export DEEPSEEK_API_KEY=your_key_here
建议

如果你只是第一次上手，先只接一个最熟悉的模型提供商，避免同时调试多套 API 配置，把问题面收窄。

第一个预测任务（5 分钟上手）

推荐直接用文档站附带的样例数据 load.csv。它包含典型的 TIMESTAMP / LOAD 字段，适合快速理解 CastClaw 的初始化、分析、审核与实验节奏。

1

准备工作目录与样例数据

把 load.csv 放到一个独立实验目录，避免不同任务复用同一份 .forecast/ 状态，便于后续回看与清理。

2

启动 TUI

在该目录中运行 castclaw。首次进入时先确认模型、实验预算与任务描述，再让 Planner 接管初始化。

3

在 Planner 页面输入任务描述

明确目标列、时间粒度、预测步长和评估指标。描述越具体，Planner 生成的分析与 Skill 草案越有针对性。

4

审核 Skill 草案

Planner 会结合定量诊断和领域分析给出 2 到 4 个候选 Skill，确认模型族、搜索空间与风险提示无明显偏差后再继续。

5

观察 Forecaster 迭代与最终报告

Forecaster 按技能执行实验循环，Critic 汇总结果并生成 final-report.md。你需要重点关注是否出现 HITL 暂停。

# 典型首次启动流程
mkdir my-first-castclaw-run
cd my-first-castclaw-run
# 将 load.csv 放入当前目录后
castclaw
Planner 输入模板

请用 load.csv 做未来 24 小时电力负荷预测；时间列为 TIMESTAMP，目标列为 LOAD，优先考虑日周期和周周期影响，评估指标使用 MAE 与 MAPE，实验预算控制在 20 次以内。

TUI 界面基础操作

TUI 不是简单的命令入口，而是你与三个智能体协同工作的控制台。理解切换逻辑和状态查看方式，能显著降低第一次使用时的混乱感。

区域 / 快捷键 作用 你应当关注什么
Ctrl+1 切换到 Planner 任务定义是否完整，分析报告与 Skill 草案是否合理。
Ctrl+2 切换到 Forecaster 实验是否停滞，当前最优结果是否持续改善，是否触发 HITL。
Ctrl+3 切换到 Critic 报告是否覆盖关键比较、性能分解与下一步建议。
任务状态面板 查看当前阶段和预算 确认当前处于 Init / Analysis / Forecasting / Report 的哪一阶段。
中断与恢复 暂停后继续当前任务 在人类反馈后恢复迭代，不需要重新初始化整个任务。
常见误区

第一次使用时最容易把 TUI 当作单步问答界面。实际上它是阶段驱动的任务工作台，重点是“当前在哪个阶段、下一步由谁负责”。

核心心智模型

核心概念

CastClaw 的核心不是某个单独模型，而是一个由运行时、Skill、插件和人工确认共同组成的流程系统。

理解这部分后，很多 CLI 和配置行为会自然变得清晰

系统架构概览

可以把 CastClaw 理解成四层协同结构：CastRuntime 负责执行与状态，CastSkill 负责策略层，插件生态承担具体能力，TimeEmbed 提供统一表征与经验对齐能力。

层级 职责 你会在什么时候接触它
CastRuntime 任务上下文管理、阶段推进、Agent Loop 执行、文件状态同步。 启动任务、切换阶段、查看预算与恢复暂停任务时。
CastSkill 根据分析结果检索或生成策略，决定模型路线、特征模板与搜索空间。 审核 Skill 草案、沉淀经验、复用历史策略时。
Plugin Ecosystem 通过 CastSense / CastFeat / CastZoo 完成诊断、表示构建和模型编排。 分析结果解释、特征设计和模型选择时。
TimeEmbed 提供跨任务表征、相似模式检索和经验对齐的能力底座。 更偏系统能力层，通常不直接手动操作，但会影响 Skill 检索质量。

Reflection / Report

三智能体协同（Planner / Forecaster / Critic）

三个智能体不是“谁更聪明”的关系，而是严格分工的流水线。你应该把它们看成不同岗位，而不是三个能互相替代的聊天窗口。

智能体 核心职责 关键行为
Planner 任务定义、数据诊断、阶段编排。 并发执行定性与定量分析，输出预测前报告，生成候选 Skill。
Forecaster 实验循环与策略试错。 读取历史、选配置、调用 CastFeat / CastZoo、记录反思、触发 HITL。
Critic 结果聚合与最终报告。 对比模型族表现、生成可视化与结构化结论，产出 final-report.md。
Planner

先看清问题，再开跑

如果 Planner 阶段定义不清，后续的实验会在错误目标上持续优化，这会比模型选错更浪费资源。

Forecaster

关注迭代节奏，不是单轮结果

Forecaster 的价值在于持续比较、反思与收敛，而不是某一次偶然跑出高分。

Critic

把实验转成结论

它负责回答“为什么这个结果可信”，而不仅仅是“哪个数字最高”。

Agentic Workflow 全流程

CastClaw 把预测任务拆成五个强约束阶段。阶段切换由工具和文件协议控制，而不是完全依赖 LLM 自觉，因此流程可追溯、可审计。

01

初始化（Init）

冻结 task.json，创建 .forecast/ 工作目录，确保所有后续实验都绑定在同一个明确任务定义上。

02

预测前分析（Pre-forecast Analysis）

双轨并发执行：一条做定性领域分析，一条做定量数据诊断，最后融合成驱动 Skill 生成的预测前报告。

03

技能审核（Skill Audit）

Planner 生成 2 到 4 个 Skill 候选后暂停，等待人工审核模型路线、风险与搜索空间，再进入实验循环。

04

预测迭代（Forecasting）

Forecaster 读取历史最佳结果，选择配置，构建特征，执行模型训练评估并产出反思记录；当持续无改善时触发 HITL。

05

后置报告（Post-forecast Report）

Critic 汇总实验产物、性能分解与可视化说明，生成结构化的 final-report.md。

流程设计重点

CastClaw 的差异化不在“跑更多模型”，而在于先分析、再审核、后迭代，并在关键节点允许人介入纠偏。

.forecast/ 目录协议

所有智能体共享状态的核心媒介就是这个目录。只要你理解了目录结构，就能理解 CastClaw 的阶段状态和实验产物落在哪。

.forecast/
├── task.json
├── skills/
├── experiments/
│   └── {run-id}/
│       ├── config.json
│       ├── result.json
│       └── reflect.md
└── reports/
    ├── pre-forecast.md
    └── final-report.md
路径 含义
task.json 冻结后的任务定义，包含目标列、步长、预算等核心上下文。
skills/ Planner 生成并由人工审核后的 Skill 文件库。
experiments/{run-id}/ 单轮实验的配置、结果与反思，供 Forecaster 与 Critic 复用。
reports/ 预测前报告和最终报告的落盘位置，是最适合人工回顾的入口。
P1 · 差异化特性

人机协作（HITL）

HITL 不是给系统兜底的“人工客服模式”，而是主动保留给研究者的高价值决策入口。

重点看：什么情况下该介入，以及介入时改什么

什么是 HITL 暂停

当 Forecaster 连续多轮无改善、出现异常结果或者当前策略与领域认知明显冲突时，系统会暂停在一个可恢复点，等待你输入反馈。这个暂停并不意味着任务失败，而是一次强制的人类校正窗口。

节点一

任务设定确认

确认目标列、时间列、预测步长、评估指标与资源约束，避免后续一切建立在错误任务定义上。

节点二

技能策略审核

确认模型族、参数搜索空间和风险提示是否合理，避免 Planner 在错误方向上生成大规模实验。

节点三

实验停滞干预

当连续无改善时，注入领域先验、限制无效模型族或补充数据特征假设，再恢复迭代。

不要把 HITL 理解成“重跑按钮”

真正有效的人工介入，应当改变策略假设，例如限定模型族、缩窄搜索空间、说明异常日期、或补充外部约束。

技能审核：如何介入

审核 Skill 时，不是看 YAML 写得是否漂亮，而是判断“这套策略是否真的适用于当前任务”。重点关注适用条件、搜索空间和风险提示三部分。

审核关注点

  • 适用条件是否与数据特征匹配，例如强季节性、长序列、频率稳定等。
  • 参数搜索空间是否过宽，是否会浪费预算在低价值区域。
  • 风险警告是否覆盖已知缺陷，例如小样本过拟合、分布漂移、夜间零值等。

建议的介入方式

  • 直接删掉明显不合适的模型族，而不是让 Forecaster 用预算去证明它不合适。
  • 把你已知的重要节假日、设备变更、政策事件写进领域说明中。
  • 缩窄学习率、窗口长度或 patch 长度等关键搜索维度。

结果确认与干预时机

最值得人工确认的不是每一轮实验结果，而是“方向性信息”发生变化的时候。例如一整类模型持续失败、最佳模型突然切换、指标改善停止，或报告结论与业务常识冲突。

连续无改善 结果与领域常识冲突 最佳模型族切换 异常日期影响明显 预算即将耗尽
一个实用原则

如果你的介入无法改变下一轮策略，就先不要介入。HITL 最有价值的地方是改变路径，而不是重复确认现状。

P2 · 能力层

插件工具箱

CastClaw 并不把“选模型”当作第一步，而是把诊断、表示和模型调用拆成可以独立演化的专业模块。

插件的职责分离，是 CastClaw 可持续扩展的关键

CastSense：数据诊断

CastSense 负责回答“这个序列现在是什么状态”。它不是只给几项统计量，而是把趋势、季节性、异常和分布变化整理成结构化知识，供 Planner 生成策略时使用。

Perception

趋势与周期识别

检测长期趋势、日周期、周周期与潜在多尺度周期，帮助判断应该优先走统计模型还是深度模型路线。

Quality

异常与漂移定位

发现突变、离群点、非平稳性和分布漂移，为风险提示与 HITL 干预提供依据。

Knowledge

结构化输出

把诊断结果沉淀为后续 Skill 检索、特征设计和模型编排都能消费的结构化 knowledge。

CastFeat：特征构建

CastFeat 负责回答“应该怎样把数据变成模型可用的表示”。它承担特征工程与表征生成，把原始时序转成更适配下游模型的 representation。

lag / rolling 统计特征 频域与多尺度表示 patch / token embedding model-ready representation
理解方式

CastFeat 并不是一个“再做一遍手工特征工程”的模块，它更像是把领域特征、统计特征和基础模型输入形式统一到同一条表示构建管线上。

CastZoo：模型编排

CastZoo 负责回答“用什么模型，以及怎么组合”。它不只是模型仓库，还负责策略化调度，包括 ensemble、pipeline 和 coarse-to-fine 等多种运行方式。

支持的模型族

统计模型如 ARIMA、ETS、Theta；机器学习模型；深度学习模型如 Informer、PatchTST；基础模型如 Chronos、TimesFM、Moirai。

支持的策略

既可以单模型直接跑，也可以做多模型集成、先粗后细的两阶段调度，或把基础模型输出作为后续模型的先验信息。

P2 · 策略层

Skill 系统

Skill 是 CastClaw 把“这次任务学到的经验”结构化保存下来的机制，它是系统能否持续进化的关键资产层。

Skill 不只是配置文件，它是可复用策略

Skill 是什么

Skill 可以理解为“经过分析与审核的策略模板”。它描述某类数据在什么条件下适合什么模型族、什么搜索空间、什么特征模板，以及有哪些已知风险。

Knowledge

承接历史经验

把一次次实验中验证过的路线沉淀下来，而不是每次从空白开始。

Policy

指导未来任务

为新任务缩小模型空间，减少盲试，让 Forecaster 一开始就站在更合理的起点上。

Audit

允许人工把关

Skill 先审核再使用，使系统进化建立在可信策略之上，而不是自动累积噪声经验。

Skill 文件结构

Skill 通常使用 YAML 表示，核心内容是适用条件、模型族、搜索空间、特征模板和风险项。下面是一个典型示例。

name: deep_learning_periodic
applicable_conditions:
  - 强季节性数据
  - 序列长度 > 5000
model_family: deep_learning
models: [PatchTST, iTransformer]
search_space:
  learning_rate: [1e-4, 5e-4]
  patch_len: [16, 32, 64]
feature_template: patch_token
risks:
  - 数据量不足时过拟合风险高
domain_notes: ""
看 Skill 时的重点

先看 applicable_conditions 和 risks，这两部分最直接决定当前任务是否该用这份 Skill。

如何审核与沉淀 Skill

Skill 的生命周期大致分四步：生成、审核、使用、沉淀。审核质量决定了系统后续是否会持续变强，还是持续放大误导。

  1. Planner 基于预测前分析草拟 2 到 4 个候选 Skill。
  2. 人工审核模型路线、适用条件和风险提示，必要时直接编辑内容。
  3. 通过审核的 Skill 进入 .forecast/skills/，供当前任务与相似任务复用。
  4. 后续随着经验积累持续演化，逐步形成团队级策略资产。
审核原则

宁可保留少量高质量 Skill，也不要囤积大量“看起来可能有用”的低信号策略。Skill 库的价值在于可信，而不是数量。

/cast-creation 命令

这个命令用于交互式生成 CAST.md 项目约束文件。适合在任务开始前明确禁用模型、预算上限、评估偏好和领域说明，让每个 Agent 都带着相同边界工作。

适合什么时候用

你已经知道哪些模型不该用、实验预算不能超过多少次，或者有必须注入给所有 Agent 的领域说明时。

它解决什么问题

避免每轮任务都口头重复约束条件，并降低 Agent 在后续阶段“忘掉限制”的概率。

P2 · 运行边界

配置参考

配置层决定系统的默认行为边界，包括预算、模型接入方式和注入给 Agent 的长期约束。

想让系统更稳，优先调约束，不要急着放大搜索空间

CAST.md 约束文件

CAST.md 是项目级行为约束，会被自动注入到 Agent 上下文中。适合承载稳定、长期、所有阶段都应该知道的规则。

字段 说明
banned_models 禁用模型列表，用于直接排除低价值或资源不允许的模型族。
max_experiments 最大实验次数，限制 Forecaster 的探索预算。
no_improve_threshold 连续无改善阈值，到达后通常触发 HITL 暂停。
eval_metric 评估指标偏好，例如 MAE、MAPE 或 RMSE。
domain_notes 领域背景说明，会注入每个 Agent 的上下文，帮助对结果做更符合业务的判断。

castclaw.json 参数

如果你希望团队共享默认模型配置和预算策略，可以使用 castclaw.json。它适合作为任务级或项目级默认配置入口。

{
  "model": "anthropic/claude-sonnet-4-6",
  "light_model": "anthropic/claude-haiku-4-5",
  "max_experiments": 20,
  "no_improve_threshold": 5
}
配置建议

把高成本、高质量模型放在主模型位，把轻量模型放在辅助分析或低成本调用路径，可以更平衡速度与质量。

模型接入（LLM 配置）

CastClaw 通过 Vercel AI SDK 风格的接入方式兼容多家模型提供商。你可以按网络环境、预算、模型偏好来选择国际或国内线路。

International

国际提供商

Anthropic Claude、OpenAI GPT、Google Gemini，适合已有稳定海外 API 接入的环境。

Domestic

国内提供商

DeepSeek、Qwen、GLM 等，适合国内网络环境和更灵活的接入策略。

Deployment

部署方式

既可以直接使用 API，也可以接自建推理服务或昇腾算力 API，便于统一企业内部部署。

P1 · 高频查询

CLI 命令参考

CLI 是 CastClaw 的主要交互入口。这里重点覆盖启动命令、TUI 快捷键，以及阶段控制相关工具。

这部分适合在使用中随查，不必一次背完

castclaw 启动选项

命令 用途
castclaw 在当前目录启动 TUI，并基于当前项目上下文接管预测任务。
castclaw --model anthropic/claude-sonnet-4-6 启动时显式指定主模型，适合临时覆盖默认配置。
castclaw --version 查看当前 CLI 版本，排查环境不一致时很有用。

快捷键速查（Ctrl+1 / 2 / 3 等）

快捷键 功能
Ctrl+1 切换到 Planner
Ctrl+2 切换到 Forecaster
Ctrl+3 切换到 Critic

把这三个快捷键用熟以后，你会自然形成一个节奏：在 Planner 定义与审核，在 Forecaster 看迭代，在 Critic 看是否能形成可信结论。

forecast_state / forecast_task 工具

forecast_state 用于阶段控制，核心意义是把流程切换变成受约束的显式动作；forecast_task 则围绕任务定义与其冻结状态展开，帮助你查看和维护当前任务上下文。

forecast_state

  • forecast_state init：初始化任务并创建 .forecast/ 目录。
  • 阶段转换强制执行，不可随意跳过，确保过程可追溯。

forecast_task

  • 面向 task.json 的任务定义辅助工具。
  • 适合在 Init 阶段确认目标列、时间列、步长与评估指标。
  • 与状态工具结合使用，可以把任务定义和阶段推进明确区分开。
P2 · 典型任务

使用示例

下面三个案例分别覆盖负荷、光伏和金融时序，目的是帮助你建立“不同数据形态对应不同策略”的感觉。

案例部分重点看数据特征、推荐 Skill 与 HITL 介入点

电力负荷预测（load.csv）

load.csv 是最适合第一个示例的入门数据。它包含小时级负荷序列，通常具有稳定的日周期和周周期，并伴随季节性峰值变化。

Data

数据特征

小时频率、约 1.5 万条样本，强日周期（24h）与周周期（168h），夏冬峰值明显。

Skill

推荐策略

优先考虑 deep_learning（PatchTST、iTransformer）加 foundation（Chronos）组合路线。

Output

预期产物

看到 pre-forecast.md、若干实验目录以及最终 final-report.md，说明主流程已经打通。

光伏发电预测

光伏序列通常同时具备强日周期、夜间恒零和天气敏感性，是非常典型的“需要领域知识”的案例。

推荐数据与诊断重点

例如 GEFCom2014 Solar Track 的小时级数据。CastSense 应重点识别夜间零值、天气突变和季节性变化。

推荐策略与 HITL

可从 statistical（Theta）+ foundation（TimesFM、Moirai）入手；HITL 重点放在阴雨连续日、天气异常日的人工标注与补充说明。

金融时序预测

金融数据高波动、非平稳，对突发事件敏感，往往不适合盲目依赖单一深度模型路线，更需要风险意识与外部事件注入。

推荐策略

更适合 statistical + foundation ensemble 的保守组合，避免把所有预算压在单一路线上。

人工介入重点

标注重大事件日期，如财报、政策发布或宏观冲击，并重点查看 CastSense 给出的分布漂移与结构断点提醒。

P3 · 按需查询

FAQ 与故障排查

常见问题聚焦在模型切换、暂停恢复、Skill 管理和环境依赖。排查时优先缩小问题范围，不要同时改多个配置面。

排查顺序：版本 → 依赖 → API Key → 配置 → 任务本身

常见问题

如何更换 LLM 提供商？

修改 castclaw.json 中的模型配置，或切换对应环境变量即可。

HITL 暂停后如何继续？

在 Forecaster 标签页输入反馈并提交，系统会在当前任务上下文中继续，而不是从头开始。

Skill 文件在哪里管理？

默认位于 .forecast/skills/ 目录，建议把审核后的稳定 Skill 再沉淀到团队共享资产库中。

为什么结果不稳定？

先检查任务定义、预算是否过小、是否存在异常日期未标注，再考虑模型和特征层面的优化。

环境问题排查

现象 排查建议
Bun 版本不足 升级到 1.3.11 或更高版本后重新打开终端，再检查 bun --version。
Python 后端报错 进入 Python 目录重新执行 uv sync，确认虚拟环境与依赖被正确安装。
API Key 未生效 检查环境变量是否已 export 到当前 shell，或确认 castclaw.json 是否覆盖了模型设置。
阶段无法推进 检查 .forecast/ 是否缺失关键文件，尤其是 task.json 和阶段相关报告产物。

这份文档站按“先上手，再理解系统，再查细节”的顺序组织。若你要第一次演示 CastClaw，建议直接从 第一个预测任务 和 Agentic Workflow 全流程 开始。
