核心判断： Agentic RL 不是把 RLHF 或 RLVR 机械地搬到多轮场景里，而是把训练问题从 单轮反馈优化 重构为 多轮、部分可观测、环境耦合的策略学习。 这会同时改变算法设计、轨迹数据结构和训练系统组织方式。

页面定位

这页不是单篇论文主页，而是一个面向整个 Agentic RL 基础算法方向 的总览入口。 它基于《A Survey of Agentic Reinforcement Learning for Large Language Models: Foundations, Paradigms, and Challenges》整理而成，重点提炼三个问题： Agentic RL 在建模上与 RLHF / RLVR 有什么本质不同、 多轮轨迹该如何表示与分配信用、 以及 为什么训练基础设施已成为算法同等重要的一部分。

在当前主页体系里，`StepPO` 和 `PSPO / PaperScout` 对应的是这个方向上的代表性方法与场景化实例； 这页本身则提供了一个更高层的基础算法视角，用来串联 credit locus、trajectory schema、 training infrastructure 和 benchmark landscape 这些共同问题。

从 RLHF / RLVR 到 Agentic RL

Survey 给出的总框架很清晰：Agentic RL 最好被理解为从 single-turn feedback optimization 走向 multi-turn environment-coupled policy learning。 也就是说，模型不再只对一个完整回答打分，而是要在带状态的环境中持续行动、观察、更新上下文。

范式 交互结构 奖励来源 主要难点
RLHF 单轮回答 人类偏好 / Reward Model 偏好建模与对齐质量
RLVR 单轮回答 可验证结果奖励 Verifier 设计与结果正确性
Agentic RL 多轮环境交互 延迟环境奖励 + 局部反馈 长程信用分配、轨迹表示、系统吞吐、环境随机性

为什么这个转变很关键

🧭

问题从 bandit 变成 POMDP

Agentic RL 更自然地被建模为 POMDP：状态不可完全观测，动作改变环境，观察在每一步到来，奖励常常延迟。

🛠️

工具调用进入策略本体

搜索、代码执行、网页点击和结构化 API 调用不再只是 workflow 模块，而成为策略必须学习的决策对象。

⏳

早期决策改变后续可见信息

一次检索、一次点击或一次错误工具调用，都会改变后续可用证据和状态空间，因此早期步骤的因果价值显著上升。

🏗️

系统不再是实现细节

Rollout-as-a-Service、异步采样、sandbox、trainer-worker decoupling 直接影响训练稳定性和可扩展性。

轨迹表示与信用分配

Survey 把 trajectory 视为 Agentic RL 的中心接口。 环境、数据结构、优化器和基础设施不是各自孤立的模块，而是都通过轨迹这个对象耦合在一起。 因此，很多算法问题其实首先是“数据应该怎样被记录和回放”的问题。

三种轨迹表示范式

表示方式 优势 主要代价
Message-based 兼容对话模板，assistant / tool / user 角色结构天然清晰 重序列化会产生 re-tokenization drift，精确 token 对齐较弱
TITO 直接回放 token id，能最大限度保持 rollout 与 training 的 token 一致性 政策 token 与环境 token 混在同一流中，必须显式做 token masking
Step-based 把交互 step 作为原子决策单位，更自然对齐环境边界与局部 credit 需要额外定义 step 边界和更结构化的 trace schema

五层信用分配视角

🔤

Token

最细粒度，能精确控制 loss scope，但高方差且强依赖轨迹是否准确区分 policy-authored spans。

🪜

Step

把推理段、工具调用段或一轮交互作为基本单位，是当前 Agentic RL 中最常见的折中方案。

🔁

Turn

适合多轮对话或协作环境，把整轮 interaction 作为奖励分配边界，常见于长上下文与协作型任务。

🎯

Trajectory

最容易扩展，常见于 GRPO 类 group-relative whole-trajectory comparison，但本地因果定位能力最弱。

👥

Agent

在多智能体系统里，除了轨迹内 credit，还要处理跨 agent 协同和团队奖励带来的额外结构。

核心难点

📏

长程信用分配

奖励通常在多轮交互后才出现，而训练更新却必须回溯到早期关键步骤，这会直接决定局部监督是否可靠。

🧩

部分可观测与环境非确定性

网页、代码仓库、检索结果和外部接口都会变化，导致相同策略在不同 rollout 中未必重现相同观察与结果。

基础算法谱系

Survey 的一个重要观点是：在 Agentic RL 中，先问“哪一个优化器最好”往往太早； 更基本的问题是 credit 被附着在哪个单位上。因此算法更适合按 trajectory-level、step-level 和 token-level 三种 credit locus 来理解。

🧱

Trajectory-Level

以 GRPO 为代表，把整条 rollout 的结果做相对比较后广播回去。优点是易扩展，缺点是难回答“到底哪一步导致成功”。

🪄

Step-Level

以 interaction step 为局部 credit 边界，在 locality 和稳健性之间取得较好平衡，是 Agent 场景下最自然的设计中心。

🎯

Token-Level

更常见的作用不是做完整 token-specific reward，而是精确定义 loss scope，避免环境 token 获得错误梯度。

🌿

Structured Exploration

Agentic RL 的探索越来越偏向有结构地在高不确定性节点分枝，而不是均匀提高采样随机性。

代表性研究页

Step-Level Optimization

StepPO

把 Agentic RL 的基本决策单位从 token 提升到 interaction step， 强调 step-level MDP、step-level credit assignment 与 step-native systems 的整体对齐，是这条基础算法线上的代表性立场论文。

查看主页 → arXiv
Sequence-Level Optimization

PSPO / PaperScout

在学术论文检索 Agent 中，把多轮 search-and-expand 过程建模为顺序决策， 通过 process-aware、sequence-level 的优化方式， 展示了场景化 Agentic RL 方法如何利用更贴近交互边界的 credit。

与当前方法页的关系

这个总览页负责回答“Agentic RL 的共同基础问题是什么”，而 StepPO 和 PSPO / PaperScout 则分别展示了 step-level 与 sequence-level 的代表性方法如何在具体任务中落地这些基础思想。

训练基础设施

Survey 特别强调，Agentic RL 的基础设施故事可以分成两个连续抬升： 第一层是从普通 LLM training / inference 升级到 LLM RL training / inference； 第二层是从 single-turn LLM RL 升级到 multi-turn agentic RL。

🔄

Rollout-as-a-Service

Rollout 从一次简单的生成调用，变成由 agent runtime、工具执行器、sandbox 和环境状态共同组成的服务系统。

⚙️

Trainer-Worker 解耦

采样和训练必须解耦扩展，才能分别优化 wall-clock-bound interaction 和 GPU-bound policy update。

🧪

Sandbox 与可复现性

代码执行、网页交互和复杂工具调用要求系统提供可控 sandbox、日志留存、轨迹重放与评估稳定性保障。

📦

Replay 与数据新鲜度

历史轨迹、warm start、negative mining 和 freshness-aware curation 直接影响训练收益与偏差累积。

为什么系统设计与算法同等重要

系统问题 影响 典型应对方向
采样极慢 多轮网页 / 代码 / GUI 交互往往比文本生成慢几个数量级 异步 rollout、批处理、优先队列、elastic workers
轨迹很长 上下文溢出、重放成本高、值函数更难稳定 压缩、截断、step-native schema、prefix reuse
环境复杂 需要真实浏览器、代码执行、状态追踪与失败恢复 sandbox、deterministic replay、标准化运行时接口
数据质量差异大 正样本稀缺，坏轨迹会显著拖累训练稳定性 quality filtering、negative mining、policy-version tracking

应用与基准

Agentic RL 正在多个任务族中扩展。真正统一这些任务的不是语义主题，而是 交互结构：Agent 需要搜集信息、采取环境动作，并在延迟反馈下改善多轮策略。

🔎

Agentic RAG / Deep Research

搜索、阅读、证据扩展与综合写作都进入同一策略循环，早期检索决策直接改变后续可用证据。

🌐

Web Navigation

动态界面、长 horizon 与脆弱评估协议让网页智能体成为最典型的 fully interactive agentic RL 场景。

💻

Coding Agents

代码编辑、测试执行和调试提供了相对可验证的回报，是 Agentic RL 当前最重要的高价值应用之一。

🖱️

GUI / Computer Use

部分可观测更强、动作空间更复杂，点击、输入、滚动和界面错误恢复都要纳入决策链条。

📊

Structured Data

结构化表格和分析任务通常具备较可验证的奖励，因此很适合研究 step-level credit 与过程奖励塑形。

👥

Collaborative Multi-Agent

一旦多智能体并发行动，除了长程 credit 之外，还要处理跨 agent 的协作、团队奖励和异步 rollout。

典型 benchmark 维度

Benchmark 家族 代表任务 Agentic RL 含义
Web WebArena、Mind2Web 检验长程交互、界面随机性和轨迹质量过滤
Coding SWE-bench、HumanEval 检验执行可验证奖励与多轮调试闭环
Open-Ended Assistant GAIA 检验异构工具、长程规划与运行时基础设施
Deep Research 多跳检索、论文搜索、证据综合 检验 retrieval、selection、synthesis 之间不均匀的 step credit

前沿问题

Survey 把开放挑战归纳为四个互相关联的问题簇。它们分别覆盖监督信号、训练起点与样本效率、 评估与训练系统，以及理论与 scaling。

🎓

学习信号与监督

如何把 trajectory-level judgment、step correction、selective oversight 与自动奖励有机组合，而不是互相冲突。

🚀

初始化与样本效率

冷启动策略很难自然碰到高奖励轨迹，如何用 staged training、curriculum 和历史轨迹 warm start 提高起步效率。

🧰

评估与训练系统

需要更稳定的环境、更可重放的工具链、更可靠的 efficiency-adjusted evaluation，而不只是单次成功率。

📐

理论与 Scaling

Agentic RL 的性能同时取决于模型规模、rollout 预算、环境复杂度和监督强度，尚缺少统一 scaling law。
